# Static Objects — Reference

How to add non-robot objects (tables, lab equipment, obstacles, downloaded 3D
models) to a generated lab so they appear in **Gazebo** and are **avoided by
MoveIt** motion planning.

The config JSON is the **single source of truth**. Its `static_objects` array is
read by `create-ros-pkg` and turned into two things:

| Output | Generated by | Purpose |
|---|---|---|
| `config/world.sdf` | `scripts/helpers/generate_world_sdf.sh` | the **Gazebo** scene (what you see) |
| `scripts/scene_publisher.py` | `scripts/helpers/generate_scene_publisher.sh` | the **MoveIt planning scene** (what the arm avoids) |

So one config entry drives both the visual sim and collision avoidance.

---

## Anatomy of an entry

Every `static_objects` entry has:
- **`name`** — unique id (also the collision-object id in MoveIt).
- **`pose`** — `[x, y, z, R, P, Y]`: position in metres + orientation in radians.
- a **geometry**: either a primitive (`box`/`cylinder`/`sphere`) **or** a model
  reference (`include` and/or `mesh`).

### 1. Primitive shapes — no files needed

These work immediately; just type them in. Avoided in MoveIt **and** rendered in
Gazebo.

```json
{ "name": "table", "pose": [0.6, 0.0, 0.25, 0, 0, 0], "box": [0.6, 0.6, 0.5] }
{ "name": "post",  "pose": [0.0, 0.5, 0.15, 0, 0, 0], "cylinder": { "radius": 0.05, "length": 0.3 } }
{ "name": "ball",  "pose": [-0.4, -0.4, 0.1, 0, 0, 0], "sphere": { "radius": 0.1 } }
```

| Field | Meaning |
|---|---|
| `box` | `[size_x, size_y, size_z]` in metres |
| `cylinder` | `{ "radius": r, "length": L }` in metres |
| `sphere` | `{ "radius": r }` in metres |

### 2. Model references — for a real 3D model

A model already present under `src/multi_arm_lab_sim_description/models/<name>/`
can be referenced two ways, usually together:

```json
{
  "name": "thermoshaker",
  "pose": [0.3, 0.5, 0.1925, 0, 0, 0],
  "include": "package://multi_arm_lab_sim_description/models/thermoshaker",
  "mesh":    "package://multi_arm_lab_sim_description/models/thermoshaker/meshes/thermoshaker.dae"
}
```

| Field | Used by | Notes |
|---|---|---|
| `include` | **Gazebo** | URI of the full model folder (`model.sdf`). Loads the model's own materials/textures, renders solid. Preferred for visuals. |
| `mesh` | **MoveIt** | path to a mesh file (`.dae`/`.stl`/`.obj`). Loaded as true triangle geometry so the arm plans around the real shape, not a bounding box. |
| `scale` | bare mesh only | optional, **and not used here**. For an `include`+`mesh` model the scale lives in the model's `model.sdf` (the single source) — see below. Only set `scale` on a bare `mesh` that has no owning model. |

- Use **both** `include` (pretty Gazebo render) and `mesh` (collision) for a
  full result. `include` alone = no MoveIt collision (warned and skipped).
- The referenced model **files must already exist** under `models/`. Getting
  them there + filling in these fields is what `scripts/add-model` automates.

## The two gotchas (why placement isn't just "set z=0")

These are per-model and are exactly what `add-model` measures for you.

### Origin / resting Z
A mesh's origin is often at its **geometric centre**, not its base. Setting
`z = 0` then buries half of it underground. Sit it on the floor with
`z = -min_z` (half its height if centred). Example: the thermoshaker is centred,
half-height `0.1925`, so it rests at `z = 0.1925`.

### Size — `model.sdf` is the single source of scale
A model's size comes from its own `model.sdf` `<mesh><scale>`. **Both renderers
read that same value**, so they can't disagree:
- **Gazebo** applies it via `include`.
- **MoveIt** (`scene_publisher.py`) looks up the mesh's owning `model.sdf` and
  applies the same scale to the collision mesh.

You therefore do **not** put `scale` in the config for an `include`+`mesh`
object. To change the size, edit the `model.sdf` (or use `add-model --scale` /
`resize-model`, which do exactly that).

On top of scale, a Collada **`<unit>`** inside a `.dae` (e.g. the cone was in
inches, `0.0254`) is applied automatically by the scene publisher — it's
intrinsic to the file, not a duplicated knob.

> The `scale` config field still exists, but only for a **bare `mesh`** (a mesh
> with no `include`, hence no `model.sdf`). There it's the sole source and is
> used by both Gazebo and MoveIt.

## Ground avoidance

MoveIt's planning scene has **no floor** by default — the arm will plan straight
through `z < 0`. The generated `scene_publisher.py` adds a large thin floor
collision box automatically. Controlled by top-level config keys:

| Key | Default | Effect |
|---|---|---|
| `avoid_ground` | `true` | add the floor collision plane |
| `ground_level` | `0.0` | floor height in metres (raise if the robot is mounted on a surface) |

The floor box top sits 2 cm below `ground_level` so a base mounted at that height
isn't flagged as already-in-collision.

> Note: `demo.launch.py` only starts `scene_publisher.py` when `static_objects`
> is non-empty, so ground avoidance currently requires at least one static object.

## The `add-model` helper

```
scripts/add-model <source> [--name N] [--scale F]
                  [--into CONFIG.json | --rabit-into RABIT.json --attach-to KEY]
```

Installs a model into `src/multi_arm_lab_sim_description/models/<name>/`, measures
it (units, size, resting Z, model.sdf scale), and emits a ready entry. Auto-detects
the source:

| Source | Detected as | Action |
|---|---|---|
| a Gazebo Fuel URL | `fuel` | downloads via `ign fuel download` (Fortress) / `gz fuel` |
| a local model folder (has `model.sdf`) | `model_dir` | copies it in as-is (keeps materials) |
| a local mesh file (`.stl`/`.dae`/`.obj`/…) | `mesh` | copies it + generates a minimal `model.sdf` wrapper |

glTF (`.glb`/`.gltf`) meshes are auto-converted to `.obj`, since Gazebo Fortress
can't render glTF directly.

Measurement needs `trimesh` (+ `pycollada` for `.dae`): `pip install trimesh pycollada`.

### Where it writes (three output modes)

| Mode | Flag | What it does |
|---|---|---|
| Print | *(none)* | prints the entry to paste yourself |
| Flat config | `--into CONFIG.json` | appends an entry to that config's `static_objects` |
| RABIT config | `--rabit-into RABIT.json --attach-to KEY` | attaches a `model: { include, mesh }` block to the `passive_object` / action-device **instance** whose `key` is `KEY` |

`--rabit-into` writes only the geometry (`include`/`mesh`); the RABIT entry already
supplies the object's name and placement (its `key` / `initial_location`). The
script prints the measured resting Z so you can set that entry's
`initial_location.z`. An unknown `KEY` fails and lists the available keys.

```bash
# attach a downloaded beaker mesh to the passive object "solution_vial" in a RABIT file
scripts/add-model ~/Downloads/beaker.glb --rabit-into configs/my_lab.json --attach-to solution_vial
```

## Resizing (one knob: `model.sdf`)

Size lives only in the model's `model.sdf`, which both Gazebo and MoveIt read — so
resizing is a single edit, no config changes:

**At add time** — `add-model --scale F` installs the model at `F` times its native
size by setting the `model.sdf` mesh scale:
```bash
scripts/add-model <url|folder|mesh> --into configs/my_lab.json --scale 2
```

**Later** — `resize-model <name> <factor>` multiplies the installed model's current
size by `factor`, editing its `model.sdf`:
```bash
scripts/resize-model thermoshaker 0.5
scripts/resize-model fuel_cone 1.5
```

Both only touch the **shared copy** under `models/`, and the change flows to every
package that references it after a rebuild. Resizing does **not** move the object —
if the new size changes its resting height, update the object's pose Z (or RABIT
`initial_location.z`) by hand. After either command, **rebuild**
`multi_arm_lab_sim_description` and relaunch.

## End-to-end workflow

```bash
# 1. add objects (or hand-edit the JSON for primitives / existing models)
scripts/add-model "https://fuel.gazebosim.org/1.0/OpenRobotics/models/Construction Cone" \
    --into configs/my_lab.json

# 2. generate the ROS package from the config
scripts/create-ros-pkg --rabit configs/my_lab.json      # or --from-json for the flat schema

# 3. build + launch
scripts/launch-wrappers/launch_my_lab
```

Confirm in the launch log:
```
added to planning scene: ['<obj>', ..., 'ground']
```

## Quick reference: how to add X

| You want to add… | Do this |
|---|---|
| A box / cylinder / sphere | Type the entry into `static_objects` directly |
| A model already under `models/` | Copy an `include`+`mesh` entry, change `name`/`pose` |
| A new model into a **flat** config | `scripts/add-model <url\|folder\|mesh> --into <config>` |
| A new model into a **RABIT** config | `scripts/add-model <url\|folder\|mesh> --rabit-into <config> --attach-to <key>` |
| A new model at a custom size | add `--scale F` to the `add-model` command |
| Resize a model you already added | `scripts/resize-model <name> <factor>` |
| The arm to avoid the floor | leave `avoid_ground` at its default (`true`) |

## Common pitfalls

- **Object sinks into / floats above the floor** → wrong `z`; use `z = -min_z`
  (helper reports it). Note `resize-model` does not adjust Z for you.
- **Right size in Gazebo, tiny/giant in RViz** → should no longer happen: both
  read scale from `model.sdf`. If it does, the mesh has no owning `model.sdf` and
  is falling back to a stale config `"scale"`.
- **Model shows in Gazebo but the arm passes through it** → it has `include` but
  no `mesh`; add a `mesh` (or a primitive) for collision.
- **Model is white / see-through in Gazebo** → the model's own `.dae`/materials
  are incomplete (e.g. the thermoshaker: declares a texture but binds no
  material, and is a non-watertight shell). This is a bad asset, not a pipeline
  issue — collision still works.
